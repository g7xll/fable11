How we built Scalar's visual model editor

Go behind the scenes with our product team on building Scalar's drag-and-drop schema editor — and what's coming next.

Fedir Davydov · 6 min read

Scalar is code-first, but not everyone wants to write a schema by hand. So we built a visual model editor that stays perfectly in sync with your code. Here's the story of how it came together.

The challenge

Most visual schema builders trap your models in a database, drifting away from your codebase. We wanted the opposite: a visual editor whose every action produces a clean, committable diff.

Two-way sync

The breakthrough was treating the canvas and the code as two views of the same source. Drag a field and we generate the same TypeScript you'd have written by hand:

generated.ts
// added via the visual editor author: fields.reference("author"),

Edit the file directly and the canvas updates instantly. Neither view is the master — the model is.

Design decisions

  • Diff-first — every drag produces a reviewable change, not hidden state.
  • Real types — the editor only offers field types the SDK actually supports.
  • Undo everywhere — backed by the same history engine as content editing.

What we learned

The hardest part wasn't the canvas — it was the merge logic that keeps code and UI consistent when both change at once. Investing there made everything else feel effortless.

What's next

We're adding model templates, relationship visualization, and live validation previews. Join the community to help shape the roadmap.

Wrap-up

The visual model editor proves code-first and visual editing aren't at odds. Thanks to everyone in the community who tested early builds and sent feedback.
