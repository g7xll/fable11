Code-first content modeling

Explore the advantages of code-first schemas and how Scalar brings structure and flexibility to your stack.

Rob Austin · 6 min read

Content modeling is the foundation of every great CMS. Get it right and your editors move fast, your developers ship confidently, and your content scales across channels. Get it wrong and you spend months untangling fields that never quite fit. Scalar takes a code-first approach to modeling — and in this post we'll explore why that matters.

What is code-first content modeling?

Code-first means your content schema lives in version-controlled code rather than being clicked together in a UI and stored in a database you can't diff. You define a model like blogPost in a TypeScript file, commit it, and your schema travels with your codebase through every branch, PR, and deployment.

  • Versioned schemas — every model change is a reviewable commit.
  • Type safety — generated types catch field mismatches at build time.
  • Reproducible environments — staging and production never drift apart.

Defining your first model

A model is just a typed object. Here is a minimal blog post schema with a reference to an author model:

models/blogPost.ts
import { defineModel, fields } from "@scalar/sdk"; export const blogPost = defineModel({ name: "blogPost", fields: { title: fields.string({ required: true }), body: fields.richText(), author: fields.reference("author"), }, });

Run scalar push and the model is live — API, GraphQL schema, and editor UI all generated automatically.

How Scalar implements the code-first approach

Under the hood, Scalar treats your model files as the single source of truth. A migration engine diffs the committed schema against the live project and applies only the changes — no destructive resets, no manual SQL.

Schema changes are always additive by default. Renaming or deleting a field requires an explicit migration so you never lose content by accident.

Benefits for different teams

For developers

Auto-generated types mean your editor autocompletes every field. Schema review happens in the same PR as the feature that needs it.

For content teams

A clean, structured editing UI is generated from the same models — no orphaned fields, no confusing legacy options.

Getting started with code-first modeling

Install the CLI, scaffold a project, and define your first model in minutes. The same workflow scales from a personal blog to a multi-team content platform.

Wrap-up

Code-first modeling brings the rigor of software engineering to content. Versioned, type-safe, and reproducible — it's how modern teams keep content infrastructure under control as they grow.
