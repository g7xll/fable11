Configuration

Tune Scalar CMS to your project — define options, connect a database, and load plugins.

Configuration file

Scalar reads its settings from a scalar.config.ts file at the root of your project. The file exports a typed config object:

scalar.config.ts
import { defineConfig } from "@scalar/core" export default defineConfig({ project: "my-cms", admin: { path: "/admin" }, database: { provider: "postgres" }, })

Run npx scalar check to validate the file against the schema before booting.

Database connection

Scalar connects to your database through a single connection string supplied via the DATABASE_URL environment variable:

.env
# PostgreSQL connection string DATABASE_URL=postgres://user:pass@localhost:5432/scalar # or SQLite for local development DATABASE_URL=file:./scalar.db
Never commit your .env file. Add it to .gitignore and use your host's secret manager for production credentials.

Environment variables

The following variables control how Scalar authenticates and runs. Defaults are applied when a value is omitted:

VariableDescriptionDefault
SCALAR_API_KEYSecret key used to sign and verify admin API requests.required
SCALAR_PROJECT_IDUnique identifier linking this instance to your Scalar workspace.required
PORTPort the development and production server listens on.3000
NODE_ENVRuntime mode that toggles caching, logging, and error detail.development

Plugins

Extend Scalar by registering plugins in the plugins array. Each plugin can hook into the admin UI, content lifecycle, or build pipeline:

scalar.config.ts
import { defineConfig } from "@scalar/core" import { seo, media } from "@scalar/plugins" export default defineConfig({ plugins: [ seo({ sitemap: true }), media({ provider: "s3" }), ], })

Advanced options

For larger deployments, Scalar exposes lower-level options to control caching, request limits, and logging:

scalar.config.ts
export default defineConfig({ cache: { ttl: "5m", strategy: "swr" }, server: { maxBodySize: "10mb" }, logging: { level: "info" }, })

See the guides for production-ready recipes covering deployment, scaling, and observability.
