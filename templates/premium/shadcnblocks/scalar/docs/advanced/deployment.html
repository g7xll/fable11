Deployment

Ship Scalar CMS to production — deploy to Vercel, run it in Docker, or self-host on your own infrastructure.

Vercel

Vercel is the fastest path to production for Scalar. Connect your Git repository and the build runs automatically, or deploy from the command line:

terminal
# install the CLI and link the project npm install -g vercel vercel link # deploy to production vercel --prod

Define your environment variables in the Vercel dashboard, or push them from a local file:

terminal
# add env vars to the production environment vercel env add SCALAR_PROJECT_ID production vercel env add SCALAR_API_KEY production vercel env add DATABASE_URL production
Set SCALAR_ENV=production on Vercel so Scalar Studio disables draft previews and enables edge caching for published content.

Docker

For containerized deployments, build an image with the official Node base. Drop this Dockerfile at the root of your project:

Dockerfile
FROM node:20-alpine AS base WORKDIR /app # install dependencies COPY package*.json ./ RUN npm ci --omit=dev # build the application COPY . . RUN npm run build # run Scalar in production mode ENV NODE_ENV=production ENV SCALAR_ENV=production EXPOSE 3000 CMD ["npm", "run", "start"]

Build and run the container locally to verify the image before pushing to a registry:

terminal
docker build -t scalar-cms . docker run -p 3000:3000 --env-file .env scalar-cms

Self-hosting

Run Scalar on your own server with a process manager and a reverse proxy. The steps below assume a fresh Linux host with Node.js installed:

  1. 1Clone and build. Pull your repository onto the host, run npm ci, then npm run build to compile the production bundle.
  2. 2Configure the environment. Create a .env file with your DATABASE_URL, API keys, and SCALAR_ENV=production. Never commit this file.
  3. 3Run with a process manager. Start the server under pm2 or systemd so it restarts on crashes and reboots: pm2 start npm --name scalar -- run start.
  4. 4Add a reverse proxy. Put Nginx or Caddy in front of port 3000 to terminate TLS and serve over HTTPS on your domain.

Environment variables

Scalar reads its configuration from the environment at boot. The following variables are recognized in every deployment target:

VariableRequiredDescription
DATABASE_URLYesConnection string for your PostgreSQL, MySQL, or SQLite database.
SCALAR_PROJECT_IDYesIdentifier for your Scalar project, used to scope content and API access.
SCALAR_API_KEYYesSecret key authenticating server-side requests to the Scalar API.
SCALAR_ENVNoRuntime mode — development or production. Defaults to development.
PORTNoPort the server listens on. Defaults to 3000.

Health checks

Scalar exposes a health endpoint so load balancers and orchestrators can verify the instance is live. Point your probe at /api/health:

terminal
# expect HTTP 200 with a JSON status payload curl -fsS http://localhost:3000/api/health # → {"status":"ok","db":"connected","uptime":1284}
The endpoint returns 503 when the database connection is down, so it doubles as a readiness gate during rolling deploys. Configure a 10-second interval with three retries before marking an instance unhealthy.
