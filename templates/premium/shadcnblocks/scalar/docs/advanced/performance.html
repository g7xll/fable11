Performance

Tune caching, CDN delivery, and query patterns to keep Scalar fast as your content and traffic grow.

Caching

Scalar caches resolved queries in a layered store. Configure the time-to-live for read responses in your scalar.config.ts so repeat requests skip the database entirely:

scalar.config.ts
export default defineConfig( { cache: { // in-memory + Redis tiered cache driver: "redis", url: process.env.REDIS_URL, // default TTL for read queries (seconds) ttl: 60, // per-collection overrides rules: { posts: { ttl: 300 } , settings: { ttl: 3600 } , }, }, } );

Mutations automatically invalidate the affected keys, so you never serve stale content after a publish.

CDN delivery

Place a CDN in front of Scalar's read API and asset endpoints to push cached responses close to your users. Scalar emits Cache-Control and ETag headers that map directly to your TTL rules, and surrogate keys let the CDN purge only what changed on each publish.

Serve images and uploads through the CDN's image pipeline rather than your origin. Scalar's asset URLs accept ?w= and ?format= params that most CDNs can transform and cache at the edge.

Query optimization

The single biggest win is selecting only the fields you render and avoiding N+1 lookups by expanding relations in one request:

query.ts
// ❌ fetches every field, then a query per author const posts = await scalar.find("posts"); // ✅ select only what you need, join authors once const posts = await scalar.find("posts", { select: ["title", "slug", "publishedAt"], expand: { author: { select: ["name"] } }, limit: 20, });

Pagination

Prefer cursor-based pagination over offset for large collections — it stays fast at any depth because the database never scans skipped rows:

paginate.ts
const page = await scalar.find("posts", { orderBy: { publishedAt: "desc" } , cursor: req.query.cursor, // opaque token from the previous page limit: 20, }); // page.nextCursor is null when there are no more results return { items: page.items, nextCursor: page.nextCursor };
