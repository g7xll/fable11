Authentication

Authenticate requests to the Scalar CMS API using API keys or OAuth 2.0, and scope each credential to exactly what it needs.

API keys

API keys are the simplest way to authenticate server-to-server requests. Create one from Settings → API Keys in Scalar Studio, give it a name, and select the scopes it should carry. Keys are shown only once at creation time, so copy the value immediately.

Pass the key in the Authorization header as a bearer token on every request:

terminal
# fetch the latest published posts curl https://api.scalar.dev/v1/content/posts \ -H "Authorization: Bearer sk_live_8f2c4a1e9b7d6f30" \ -H "Content-Type: application/json"
Never commit API keys to version control or ship them in client-side bundles. Store them in environment variables or a secrets manager, and rotate any key that may have been exposed.

Scopes & permissions

Every credential carries a set of scopes that determine which resources it can read or modify. Grant the narrowest set of scopes that your integration requires:

Scope Description
content:read Read published and draft content entries, collections, and assets.
content:write Create, update, publish, and delete content entries and assets.
models:write Define and modify content models, fields, and validation rules.
admin Full access including members, API keys, webhooks, and billing settings.

OAuth 2.0

For apps acting on behalf of a user, use the OAuth 2.0 authorization code flow. Register your application to receive a client_id and client_secret, then redirect the user to the Scalar authorization endpoint:

http
# 1. redirect the user to authorize GET https://auth.scalar.dev/oauth/authorize ?response_type=code &client_id="app_3a9f2c" &redirect_uri="https://yourapp.com/callback" &scope="content:read content:write" # 2. exchange the returned code for an access token curl -X POST https://auth.scalar.dev/oauth/token \ -d grant_type=authorization_code \ -d code="ac_7d1e9b..." \ -d client_id="app_3a9f2c" \ -d client_secret="cs_live_..."

The token response includes an access_token, a refresh_token, and an expires_in value in seconds.

Token refresh

Access tokens expire after one hour. Use the long-lived refresh token to obtain a new access token without prompting the user again:

terminal
curl -X POST https://auth.scalar.dev/oauth/token \ -d grant_type=refresh_token \ -d refresh_token="rt_5c2a..." \ -d client_id="app_3a9f2c" \ -d client_secret="cs_live_..." # → { "access_token": "...", "expires_in": 3600 }

Security best practices

Follow these guidelines to keep your Scalar credentials safe:

  • Store keys and secrets in environment variables or a dedicated secrets manager, never in source control.
  • Issue separate keys per environment and per integration so they can be revoked independently.
  • Grant the minimum scopes required, and avoid the admin scope for routine integrations.
  • Rotate keys on a regular schedule and immediately after any suspected exposure.
  • Always make API requests over HTTPS so credentials are never sent in plaintext.
